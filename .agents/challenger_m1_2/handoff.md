# Handoff Report — Challenger 2 (Milestone 1)

## 1. Observation

During empirical stress testing of the pricing helper (`src/lib/pricing.ts`) and mock AliExpress client (`src/lib/aliexpress-mock.ts`), the following specific issues and behaviors were observed:

### Observation A: Duplicate SKU Generation in Mock Client
When executing a loop to import 200 arbitrary product IDs using `mockImportAliExpressProduct`, several products generated duplicate SKUs among their variants:
* **Product ID**: `10050010000007`
* **Colors generated**: `["indigo", "indigo"]`
* **Resulting SKUs**: 
  ```json
  [
    "MOCK-10050-IND-M",
    "MOCK-10050-IND-L",
    "MOCK-10050-IND-XL",
    "MOCK-10050-IND-M",
    "MOCK-10050-IND-L",
    "MOCK-10050-IND-XL"
  ]
  ```
* **Command executed**: `node ./node_modules/vitest/vitest.mjs run src/lib/__tests__/challenger.test.ts`
* **Output**:
  ```
  Found 6 products with colliding SKUs out of 200 tested.
  ```

### Observation B: Seed Fallback Bug for Specific Numeric IDs
In `src/lib/aliexpress-mock.ts`, the seed logic is:
```typescript
const numericSeed = parseInt(id.slice(-8)) || 99999
```
* For ID `10050000000000` (ends in 8 zeros), `parseInt("00000000")` evaluates to `0`. Due to the falsy fallback `|| 99999`, the seed is set to `99999`.
* For invalid non-numeric ID `abcdefgh`, `parseInt` evaluates to `NaN`, which also falls back to `99999`.
* Comparing the products imported for `10050000000000` and `abcdefgh` shows they are identical in title, description, and pricing options (clashing).

### Observation C: Extremely Loose URL/ID Parsing
In `src/lib/aliexpress-mock.ts` (lines 152-160):
```typescript
export function parseProductUrl(url: string): string | null {
  const cleanUrl = url.trim()
  const match = cleanUrl.match(/\/item\/(\w+)\.html/)
  if (match) return match[1]

  // Also match alphanumeric strings
  if (/^\w+$/.test(cleanUrl)) return cleanUrl
  return null
}
```
* Callers passing single-word alphanumeric strings like `invalid_url_no_match` bypass validation and are treated as valid AliExpress IDs.

### Observation D: Lack of Negative Price Validation
In `src/lib/pricing.ts` (lines 11-32):
* The helper `calculateRetailPrice` does not guard against returning negative prices:
  `calculateRetailPrice(1000, { markupType: 'fixed', fixedMarkup: 500, marginMultiplier: 1.5 }, 'fixed', -1200)` returns `-200` cents.
  `calculateRetailPrice(1000, { markupType: 'fixed', fixedMarkup: 500, marginMultiplier: 1.5 }, 'multiplier', -0.5)` returns `-500` cents.

### Observation E: Fractional Cents Retention in Fixed Markup
In `src/lib/pricing.ts` (lines 19-24):
* If `rawPriceCents` is a float (e.g. `100.4`), the returned retail price is `100.4 + fixedMarkup`, retaining decimal cents instead of rounding to an integer cent value.

### Observation F: TanStack Router Mock Error in Global Test Suite
When executing the full test suite (`node ./node_modules/vitest/vitest.mjs run`), the test runner fails inside `tests/e2e/tier2.test.tsx`:
* **Error**: `No "createRootRouteWithContext" export is defined on the "@tanstack/react-router" mock`
* **Trigger**: `tests/e2e/tier2.test.tsx` (line 18) imports `THEME_INIT_SCRIPT` from `src/routes/__root.tsx`, which evaluates `createRootRouteWithContext`. However, the mock in `tests/setup.ts` lacks this method.

## 2. Logic Chain

1. **Vulnerability 1 (Duplicate SKUs)**: In `src/lib/aliexpress-mock.ts`, the list of colors is generated by calling `faker.color.human()` twice consecutively without checking for uniqueness. This makes color duplication possible (Observation A). Additionally, the SKU formula `color.toUpperCase().substring(0, 3)` truncates color names, meaning distinct colors starting with the same prefix (e.g. `"grey"` and `"green"`) generate identical SKU prefixes. Because combinations loop over these values, duplicate SKUs are outputted within the same product model.
2. **Vulnerability 2 (Seed Collision)**: The seed logic in `src/lib/aliexpress-mock.ts` uses the boolean OR fallback operator `|| 99999` (Observation B). Because `0` is falsy, numeric IDs ending with eight zeros (which parse to `0`) and non-numeric invalid IDs (which parse to `NaN`) both fallback to seed `99999`. This causes distinct identifiers to yield identical products, breaking the mock's determinism and uniqueness assumptions.
3. **Vulnerability 3 (Weak URL Parser)**: The regex `/^\w+$/` matches any word containing letters, numbers, and underscores (Observation C). Because of this, standard words are accepted as valid IDs, delaying validation failures.
4. **Vulnerability 4 (Negative Storefront Prices)**: The function `calculateRetailPrice` only does a direct mathematical calculation (`+` or `*`) without bounding check (Observation D). This allows customized settings or product-level overrides containing negative values to slip through and result in negative storefront prices.
5. **Vulnerability 5 (Float Propagation)**: The fixed markup branch returns `rawPriceCents + fixedAdd` without applying `Math.round()` on the total (Observation E), leading to float values.
6. **Vulnerability 6 (Global Test Crash)**: The `@tanstack/react-router` mock in `tests/setup.ts` does not implement `createRootRouteWithContext` (Observation F). Since `tests/e2e/tier2.test.tsx` imports from `src/routes/__root.tsx`, the route evaluation crashes the test suite.

---

## 3. Caveats

* The tests were run in the JSDOM SQLite environment, which behaves identically to real SQLite in terms of transactions, but real-world concurrency issues in Cloud environments were not simulated.
* It is assumed that the database schema will reject duplicate SKUs on product variant tables (which is correct due to the primary key constraint on `sku`). This means the duplicate SKU generation bug in the mock client will cause database insertions to throw exceptions rather than silently corrupting data.

---

## 4. Conclusion

1. The **Mock AliExpress Client** contains bugs:
   * **Vulnerability (High)**: Duplicate SKU generation due to missing color uniqueness checks and overlapping 3-letter color prefixes.
   * **Bug (Medium)**: Seed collision for IDs ending in eight zeros (they resolve to seed `99999`).
   * **Bug (Low)**: Too permissive URL parser accepting arbitrary alphanumeric strings.
2. The **Pricing Helper** lacks input bounds checking:
   * **Vulnerability (Medium)**: Returns negative storefront retail prices if fixed markups or multipliers are negative.
   * **Bug (Low)**: Fractional cents are not rounded in fixed markup calculation.
3. The **Test Infrastructure** is incomplete:
   * **Bug (High)**: `@tanstack/react-router` mock lacks `createRootRouteWithContext`, causing E2E tests in `tier2.test.tsx` to crash.

Run the dedicated Challenger test file using:
```bash
node ./node_modules/vitest/vitest.mjs run src/lib/__tests__/challenger.test.ts
```

### Invalidation Conditions:
* If the mock client is modified to ensure unique colors and unique SKU generation, the SKU collision test will fail (or report 0 collisions).
* If the mock client's seed logic is corrected to handle `0` properly (e.g. `numericSeed = isNaN(parsed) ? 99999 : parsed`), `pZero.title` will no longer equal `pExplicit99999.title`.
* If `@tanstack/react-router` mock is updated with `createRootRouteWithContext`, the global E2E tests run will complete without the router import crash.
